## 1. LEAN 사상 — 칸반의 뿌리

칸반은 LEAN이라는 더 거대한 사상 위에 서 있다.

- **핵심 목표** : 고객이 돈을 주고 구입하는 **Value(가치)** 를 딜리버리하는 것
- **방법** : Non-value adding 활동을 제거하는 것

### 3가지 낭비 요소 (3M)

| 개념 | 의미 | 세부 내용 |
|------|------|-----------|
| **Muda** (낭비) | 가치를 만들지 않는 활동 | - **Necessary Waste** : 필요하지만 최소화해야 함 (Testing, Planning, Reporting) <br> - **Pure Waste** : 완전히 없애야 함 (불필요한 Meeting 등) <br> - **Context Switching** : 여러 일을 동시에 하면 Loss 발생 → 없애야 할 Muda |
| **Mura** (불균형) | 일을 일정하게 하지 않는 것 | - 무슨 일이 일어날지 예측 불가 <br> - 이븐하게 일을 관리하는 것이 안정적 → **칸반으로 해결** |
| **Muri** (과부하) | 일을 과하게 하는 것 | - Over demanding <br> - Lack of training / communication / proper tools |

> **Muri를 파악** → **Mura를 최적화** → **Muda를 제거** 하는 순서로 접근한다.

## 2. 칸반(Kanban)이란?

- **어원** : 한자(Kanji) = 보드(Board) / 히라가나 = 시그널 카드(Signal Card)
- **본질** : Knowledge Work(지식 노동)에 대한 매니지먼트 시스템
- **핵심 메커니즘** : Pull 시스템 — Demand와 Capacity의 밸런스를 맞추는 것
- **주의** : 칸반을 쓴다고 Throughput(처리량)이 자동으로 늘지는 않는다

> [!tip] KSD란?
> 칸반에서 다루는 **업스트림(Upstream)** 영역

### 칸반의 기본 철학

- 모든 시스템 개발에서 칸반을 사용할 수 있다. 어떻게 적용할 것인가? → **창의력 필요**
- 대부분의 회사가 끝이 아닌 **시작에 포커스**를 둔다 (시작만 하고 보여주기식 일)
	- 칸반은 반대로 **완료(Right to Left)**를 강조한다
- WIP 제한을 도입할 때 **매니저의 저항**이 발생할 수 있다 → 창의적으로 해결해야 함
- 100개의 팀이 있다면 100개의 다른 칸반이 나온다. 복사 붙여넣기식이 아닌, **우리 팀만의 맞춤형 디자인**이 필요하다.

## 3. 칸반 3대 원칙 체계

### 변화 관리 원칙 (Change Management Principles)

1. **현재 하고 있는 것부터 시작하라** *(Start with what you do now)*
   - 현재 실제로 실행 중인 프로세스를 이해하고, 기존의 역할·책임·직함을 존중한다.
2. **진화적 변화를 통한 개선에 합의하라**
   - 혁명적 전환이 아닌 점진적·진화적 변화를 추구하기로 조직이 합의한다.
3. **모든 직급에서의 리더십 행동을 장려하라**
   - 특정 직급에 국한하지 않고 조직 내 모든 계층이 리더십을 발휘하도록 장려한다.

> [!tip] 왜 "현재에서 출발"인가?
> XIT 사례에서 PSP/TSP 방법론 변경 불가라는 제약이 오히려 이 원칙을 강제했고, 결과적으로 저항 없는 도입을 가능하게 했다. 제약이 역설적 행운이 된 셈.

### 서비스 전달 원칙 (Service Delivery Principles)

> 조직은 **정책(Policies)** 으로 연결된 **상호 의존적 서비스들의 네트워크** 다.

1. **고객의 니즈와 기대치를 이해하고 집중하라**
2. **작업을 관리하라. 작업자들이 그 주위에서 스스로 조직화하게 하라**
   - *"Manage the work, not the people"*
3. **네트워크와 정책을 정기적으로 검토하여 성과를 개선하라**

> [!note] 핵심 관점
> 각 부서의 **부분 최적화**보다 고객에게 전달되는 **전체 최적화(Whole Optimization)** 가 중요하다.

## 4. 칸반 6가지 일반 실천법 (General Practices)

| # | 실천법 | 설명 |
|---|--------|------|
| 1 | **Visualize** (시각화) | 작업과 프로세스를 눈에 보이게 만든다 |
| 2 | **Limit WIP** (재공품 제한) | 동시에 진행 중인 작업 수를 제한한다 |
| 3 | **Manage Flow** (흐름 관리) | 작업이 막힘없이 흘러가는지 관찰·조절한다 |
| 4 | **Make Policies Explicit** (정책 명시화) | 규칙을 누구나 볼 수 있게 공개한다 |
| 5 | **Implement Feedback Loops** (피드백 루프) | 정기적으로 검토하고 소통한다 |
| 6 | **Improve Collaboratively, Evolve Experimentally** | 과학적 방법론으로 협력하여 발전시킨다 |

### 실천법 상세

#### Visualize — 시각화

> *"보여줄 수 없는 일은 존재하지 않는다."*

- **칸반 보드** : 작업과 프로세스를 시각화하는 대표적 도구
- 칸반 시스템 필수 요소
  - **Commitment Point** (약속 지점) 정의
  - **Delivery Point** (인도 지점) 정의
  - **WIP 제한** 표시

#### Limit WIP — WIP 제한 & Pull 시스템

> *"밸런스를 맞출 수 있는 장치 = WIP Limit"*

- WIP 제한 도입 → **Push 시스템 → Pull 시스템** 으로 전환
- **Pull 시스템** : Demand가 존재하고, 처리할 **Capacity가 있을 때만** 작업을 스케줄링
- WIP Limit = 가용 Capacity를 나타내는 신호
- 빈 슬롯(열 위의 숫자나 빈 공간) 자체가 새 작업을 당겨오라는 **신호**

> [!note] Push vs Pull
> **Push 시스템**은 수요가 처리 능력을 초과해도 계속 일을 밀어 넣는다. 결과적으로 WIP가 쌓이고 리드 타임이 폭증한다. **Pull 시스템**은 WIP 슬롯이 비어야만 새 작업을 당겨온다. 이 단순한 차이가 예측 가능성을 만든다.

#### Manage Flow — 흐름 관리

- 작업 흐름은 **가치 전달 최대화** + **Lead Time 최소화** + **예측 가능성** 을 목표로 함

**옵션과 약속 (Options & Commitment)**

- 칸반은 **Commitment(약속)를 최대한 미룬다**
- 작업 항목은 '옵션'으로 간주 → 가능한 한 늦게 선택하거나 폐기
- 마지막 순간까지 정보를 수집한 뒤 가장 현명한 선택을 함

**옵션 폐기율 (Option Discard Rate)**

- 아이디어 단계에서 버려지는 비율은 생각보다 높음 *(예: Microsoft IT — 폐기율 48%)*
- 폐기율 0% = 불확실성을 전혀 고려하지 않는다는 위험 신호
- 단, **Active 상태에서 취소되는 비율은 낮아야** 생산성이 높아짐

**2단계 약속 (2-Phase Commit)**

| 단계 | 시점 | 내용 |
|------|------|------|
| **1st Commitment Point** | 아이디어 → Dev Ready 대기열 진입 시 | 개발 시작을 결정 |
| **2nd Commitment Point** | UAT 완료 후 배포 직전 | 고객에게 특정 출시 날짜를 약속 |

#### Make Policies Explicit — 정책 명시화

> *"Flow를 만들기 위한 조건"*

- 규칙을 명확히 정의 → 팀원이 스스로 결정할 수 있는 **Empowerment** 생성
- **좋은 정책의 4가지 특징**
  - Simple (심플)
  - Visible (누구나 볼 수 있음)
  - Always applied (항상 적용)
  - **Readily changeable (언제든 바꿀 수 있음)**

## 5. STATIK: 칸반 도입을 위한 8단계 전략

칸반은 단순히 보드를 만드는 것이 아니라, 조직을 하나의 **서비스 네트워크**로 이해하는 것에서 시작한다.  
모든 칸반의 적용은 **STATIK(Systems Thinking Approach to Introducing Kanban)**으로부터 시작한다.

> 고객이 누구인가? **서비스 제공자(Provider)**와 **엔드유저(End User)**를 명확히 구분하는 것에서 시작한다.

> [!note] STATIK은 선형이 아닌 순환 구조다
> 8단계를 한 번 완료하는 것으로 끝나지 않는다. 실행 후 다시 불만족을 재분석하고, 시스템을 지속적으로 진화시키는 **반복적(Iterative) 프로세스**다.

```
① Fit for Purpose
      ↓
② Sources of Dissatisfaction
      ↓
③ Analyze Demand
      ↓
④ Analyze Capability
      ↓
⑤ Model the Workflow
      ↓
⑥ Identify Classes of Service
      ↓
⑦ Design the Kanban System
      ↓
⑧ Negotiate Implementation
      ↓
   (다시 ②로 순환)
```

1. **서비스 목적 파악 (Fit for Purpose)**
	- 제공되는 서비스가 고객에게 맞는가?
	- 고객이 우리에게 기대하는 가치가 무엇인지 정의한다.
2. **불만족의 근원 분석 (Sources of Dissatisfaction)**
	- 컴플레인(Complaint)과 불만족(Dissatisfaction)을 구분해야 한다 → 컴플레인은 액션 아이템으로 연결되지 않는다.
	- **불만이 없으면 칸반 적용의 명분이 없다.**
	- 내부(Internal): 팀원들의 과부하, 비효율 등
	- 외부(External): 느린 배송, 예측 불가능한 일정 등 고객의 불만
3. **수요 분석 (Analyze Demand)**
	- 어떤 종류의 일이, 어떤 경로로, 얼마나 들어오는지 **숫자**로 파악한다.
	- Work Item Type별로 수요의 구성이 무엇인지 분석해야 한다.
4. **전달 능력 분석 (Analyze Capability)**
	- 우리 팀이 실제로 처리할 수 있는 양을 데이터(Throughput)로 확인한다.
	- 수요가 처리 능력을 초과하는 **Push 시스템**의 상태를 파악하고, 티켓을 가용량에 맞춰야 한다.
5. **워크플로우 모델링 (Model the Workflow)**
	- 가치 전달 단계를 시각화한다.
	- 단순한 인수인계(Handoff) 기준이 아닌 **'지식 발견(Knowledge Discovery)'** 단계를 기준으로 재설계하는 것이 핵심이다.
6. **서비스 클래스(CoS) 정의 (Identify Classes of Service)**
	- 일의 성격(긴급, 표준, 날짜 고정 등)에 따른 처리 규칙을 정한다.
7. **칸반 시스템 디자인 (Design the Kanban System)**
	- 보드 레이아웃, WIP 제한, 가용성 할당을 설계한다.
8. **합의 및 실행 (Negotiate Implementation)**
	- 이해관계자들과 새로운 시스템 운영 방식을 공유하고 협상한다.

## 6. 서비스 클래스 (Classes of Service, CoS)

모든 일의 우선순위를 '긴급'으로 두지 않고, **지연 비용(Cost of Delay)** 의 성격에 따라 다르게 대우하는 방식이다.

> [!tip] CoS 분류 기준 질문
> *"What will happen if you don't finish this on time?"*
> → 결과(Impact)의 크기와 성격에 따라 클래스가 결정됨

| 클래스                    | 특징                                  | Cost of Delay 곡선 | 예시               |
| ---------------------- | ----------------------------------- | ---------------- | ---------------- |
| **Expedite** (긴급)      | 즉시 처리, WIP limit = 1, 다른 모든 작업보다 우선 | 매우 높고 즉각적        | 서버 다운, 시스템 장애    |
| **Fixed Date** (날짜 고정) | 특정 기한이 정해진 작업, 날짜를 넘기면 가치 급락 또는 위약금 | 날짜 이후 급격히 상승     | 세금 신고 마감, 법령 시행일 |
| **Standard** (표준)      | 일반적인 FIFO 처리, 비즈니스 가치를 지닌 작업        | 선형적으로 증가         | 기능 개선, 일반 버그 수정  |
| **Intangible** (무형)    | 당장 급하지 않으나 방치하면 장기적으로 누적            | 낮지만 장기적으로 누적     | 기술 부채 해소, 리팩토링   |

> [!example] XIT 사례 — CoS 식별
> 시스템을 마비시키던 PTC(Production Text Changes)를 세분화:
> - 진짜 급한 것 → **긴급(Expedite)**
> - 세금 관련 업데이트 등 기한이 정해진 것 → **고정일(Fixed Date)**
>
> 이를 통해 무분별한 "모두 최우선" 처리에서 벗어나 각 클래스에 명시적 정책을 부여할 수 있었다.

## 7. 변동성(Variability) 관리와 예측 가능성

칸반의 목표는 불확실한 상황에서도 **예측 가능한 흐름**을 만드는 것이다.

### 추정(Estimation) 대신 보충(Replenishment)

- 정확하지 않은 시간을 계산하는 데 에너지를 쓰기보다, WIP 제한 내에서 **적기 처리(Just-in-Time)**를 통해 유연하게 대응한다.
- 추정치는 종종 불만족의 원인이 된다. 그 시간을 실제 작업이나 SLA 합의에 쓰는 것이 낫다.
- 스크럼은 예측(Estimate) 기반 → 그래서 틀린다. **칸반은 서비스 딜리버리 모델**이다.

### 변동성의 원천

- **예측(Estimates)**: 추정치는 가변적이므로, 큐 보충(Queue Replenishment)으로 대체하여 유연성을 확보한다.
- **계획(Planning)**: 고정된 계획 대신 SLA를 통해 지속적으로 재계획한다.
- **폭주하는 트래픽(PTC Arrival Rate)**: 갑자기 몰리는 일은 WIP 제한과 Just-in-Time 방식으로 완화한다.

### 배치 사이즈(Batch Size)

- 배치 사이즈가 커지면 모든 것이 느려진다 → SLA를 늘리는 악영향을 미친다.

## 8. 피드백 루프 — Cadence (케이던스)

칸반에서 **사이클(리듬)을 부여하는 것** = **Cadence(케이던스)**

각 이벤트마다 독립적인 케이던스를 가질 수 있다는 점이 스크럼과의 핵심 차이다.

> [!tip] 스크럼 vs 칸반
> - **스크럼** → 스프린트라는 **하나의 고정된 사이클**에 모든 이벤트를 맞춤
> - **칸반** → 이벤트마다 **각자의 케이던스**를 가짐 → 더 유연하고 현실에 맞게 조정 가능

### 3가지 주요 케이던스

| 케이던스 | 빈도 | 목적 |
|---|---|---|
| **Replenishment Meeting** (보충 미팅) | 주간 | 빈 슬롯에 당겨올 작업을 선택해 큐를 채움 |
| **Daily Kanban Meeting** (칸반 미팅) | 매일, 10~20분 | 보드 오른쪽→왼쪽으로 흐름 점검, 블로커 식별 |
| **Flow Review** (흐름 검토) | 주간/월간 | 리드 타임·Throughput 등 데이터 기반 개선 논의 |

#### Replenishment Meeting (워크플로우 보충 미팅)
- **목적**: 다음 단계로 당겨올 작업 요청을 선택하고 큐(Queue)를 채움
- **XIT 적용**: 월간 계획 회의 → 주간 15~20분 전화 회의로 대체
  - 핵심 질문: *"다음 25일 내 납품을 원하는 항목 3개는 무엇인가?"*
  - 이후 자동화: 빈 슬롯 발생 시 자동 이메일 → 2시간 내 보충

#### Daily Kanban Meeting (칸반 미팅)
- **시간**: 10~20분 (짧게 유지)
- **방향**: 보드의 **오른쪽(완료)에서 왼쪽(시작)** 으로 이동하며 흐름 점검
- **핵심**: 작업자가 아닌 **'작업(Work)'** 에 대해 논의
- 막힌(Blocked) 항목을 식별하고 해결

#### Flow Review (흐름 검토)
- **데이터 기반**: 리드 타임, 도착률(Arrival Rate), 전달률(Throughput) 등 정량적 지표 활용
- 목표 달성 여부 확인 및 개선 사항 논의

## 9. 가용성 할당 (Capacity Allocation)

팀의 에너지를 100%로 가정하고, 서비스 유형별로 비율을 **전략적으로 배분**한다.

- **수동적 가용성 할당**: 들어오는 대로 처리
- **전략적 가용성 할당**: 유형별 비율을 사전에 정의
	- 예시: 신규 기능 50% / 버그 수정 20% / 기술 부채 20% / 긴급 대응 10%
- 이는 특정 업무가 시스템을 점령하여 **터지는 것**을 막는 안전장치다.

### 성숙도(Maturity Level)에 따른 변화

- ML 낮을수록 WIP를 높임 (전체 홀드)
- ML 올라갈수록 케이던스(미팅)가 늘어남

## 9. 실무 보드 설계 및 시각화 기법

보드는 팀의 업무 모델을 가장 잘 반영하는 **시각적 도구**여야 한다.  
처음부터 복잡하게 만들지 말고, **간단한 스텝으로 시작**하는 것이 가장 좋다. 디자인은 계속 유연하게 변해야 하며, 결코 "완성"이란 없다.

### 시각화 장치

| 요소 | 설명 |
|------|------|
| **스윔레인 (Swim Lanes)** | 가로 줄을 통해 작업 유형, 서비스 클래스, 요청자별, 프로젝트별로 구역을 나눠 가용성(Capacity)을 격리한다 |
| **티켓 색상** | 서비스 클래스나 조직 단위를 구분하며, 보드 전체의 가용성 할당 상태를 한눈에 보여준다 |
| **티켓 나이 (Ticket Age)** | 작업이 시스템에 머문 시간을 점(dot)이나 숫자로 표시하여 Pull 결정의 근거로 삼는다. 오래된 일을 먼저 해결하는 기준이 된다 |
| **아바타 (Avatars)** | 누가 어떤 일을 하고 있는지 사진이나 기호로 표시하여 책임 소재를 명확히 한다 |
| **타임스탬프** | 리드 타임 측정을 위한 시작/종료 시각 기록 |
| **블로커 표시** | 막힌(Blocked) 항목을 즉시 식별 가능하도록 표기 |
| **풀 기준 (Pull Criteria)** | 다음 단계로 넘어가기 위한 최소한의 품질 요건을 컬럼 상/하단에 명시한다. **꼭 정의해두어야 한다.** |

> [!note] 티켓에 담겨야 할 정보
> Pull 결정(당겨오기)에 필요한 정보가 티켓 안에 충분히 있어야 한다. 정보가 부족하면 구두 확인이 필요해지고, 이는 흐름을 방해한다.

### 특수 상황 모델링

- **단기 활동 (Short-Lived Activities)**
	- 너무 빨리 끝나서 별도 컬럼을 만들기 모호한 작업(예: 승인, 배포)은 컬럼 사이의 **"선 위(On the line)"**에 체크박스 등으로 표시한다.
- **비순차적 활동 (Non-sequential)**
	- 순서가 고정되지 않은 작업은 티켓 안에 **체크리스트**를 넣어 관리하면 보드 레이아웃을 단순하게 유지할 수 있다.
- **분리 및 재결합 (Split and Recombine)**
	- 하나의 큰 작업이 여러 태스크로 나뉘었다가 다시 합쳐지는 패턴을 모델링할 때, WIP 제한이 꼬이지 않도록 주의해야 한다.

### 장애물 및 외부 의존성 관리

- **블로킹 이슈 (Blocking Issues)**
	- 작업이 멈추면 티켓 위에 별도의 포스트잇을 붙이고, **원인과 발생 시간**을 기록한다.
	- 예상하지 못한 변수이며, 이를 관리하는 것이 **리스크 매니지먼트의 핵심**이다.
- **외부 의존성 (External Dependencies)**
	- 우리 팀 통제 밖의 일(벤더 대기 등)은 별도의 대기 구역(Waiting Area)에 두되, **SLA 시계는 계속 가고 있음**을 점(dot) 등으로 표시하여 잊히지 않게 관리한다.

### Upstream vs Downstream Kanban

- **Upstream Kanban**: 아직 시작하지 않은 일의 큐를 관리하는 영역. 무엇을 언제 시작할지 결정한다.
- **Downstream Kanban**: 이미 시작된 일의 흐름을 관리하는 영역. 어떻게 완료까지 끌고 갈지 관리한다.
- **Replenishment Meeting**은 이 두 영역을 연결하는 지점이다. Upstream에서 선택된 항목이 Downstream으로 넘어오는 순간이 **Commitment Point**다.

## 10. 흐름 분석 지표 (Metrics)

숫자를 근거로 대화하고, 정기적으로 시스템을 점검한다.

### 리틀의 법칙 (Little's Law)

$$\text{Lead Time} = \frac{\text{WIP}}{\text{Throughput}}$$

- 리드 타임을 줄이려면 사람을 쥐어짜는 것이 아니라 **WIP를 줄이는 것**이 핵심이다.

### 흐름 효율성 (Flow Efficiency)

$$\text{Flow Efficiency} = \frac{\text{Work Time}}{\text{Lead Time}} \times 100$$

- **Work Time** : 실제로 작업이 진행되는 순수 작업 시간
- **Lead Time** : 요청부터 완료까지 총 시간 (작업 시간 + **대기 시간**)
- 업계 평균 흐름 효율성은 **15~20% 수준**에 불과하다. 나머지 80~85%는 대기(Wait) 또는 낭비다.
- 개선 방법 : 작업 속도 향상보다 **대기 시간(Waste) 제거**에 집중

> [!info] XIT 사례의 흐름 효율성
> 평균 엔지니어링 시간 11일 / 리드 타임 125~155일 ≈ **약 8%**
> 나머지 92%는 대기 또는 낭비. 개선 잠재력이 그만큼 크다는 의미.

### CFD (Cumulative Flow Diagram)

시간의 흐름에 따른 시스템 상태를 한눈에 파악하는 핵심 차트.

- **수요(Demand)** vs. **처리 역량(Capability)** 의 격차 시각화
- WIP 수준, 대략적인 리드 타임, 전달률(Delivery Rate) 파악
- **밴드가 두꺼워지면** → WIP 과다 신호
- **밴드가 평평해지면** → 처리 중단(병목) 신호

### Lead Time Distribution (히스토그램)

- 리드 타임 데이터의 분포를 통해 실제 역량과 SLA 기대치를 비교한다.
- **85th percentile**을 SLA 기준으로 활용하는 것이 일반적이다. 평균값은 이상치에 왜곡되기 쉽기 때문이다.
- XIT 사례: 엔지니어링 노력 분포 히스토그램으로 평균 비용을 도출 → 견적 대체 근거로 활용

### 주요 지표 요약

| 지표 | 설명 | 방향 |
|------|------|------|
| **Lead Time** | 요청부터 완료까지 총 시간 | ↓ 낮을수록 좋음 |
| **Throughput (Delivery Rate)** | 단위 시간당 완료된 항목 수 | ↑ 높을수록 좋음 |
| **Flow Efficiency** | 리드 타임 중 실제 작업 시간의 비율 | ↑ 높을수록 좋음 |
| **Failure Demand** | 재작업 (제대로 안 해서 다시 하는 일) | ↓ 최소화 |
| **Blockers** | 작업 흐름을 막는 사유 | ↓ 제거 대상 |
| **WIP Levels** | 현재 시스템 내 총 작업 수 | ↓ 제한 필요 |

> 티켓에 기록되는 '나이'와 '블로킹 원인'은 나중에 **흐름 검토(Flow Review)** 시 개선의 핵심 근거 데이터가 된다.

## 11. 피드백 루프 (Cadences)

칸반 시스템을 유지·발전시키기 위한 정기 미팅 체계. 각 이벤트는 각자의 리듬(Cadence)을 가진다.

> [!note] ML 단계와 케이던스
> - ML 낮을수록 기본 3가지 케이던스 위주로 운영
> - ML 올라갈수록 케이던스가 늘어나며 시스템이 정교해짐

### 기본 3가지 Cadences

| 미팅 | 주기 | 소요 시간 | 목적 |
|------|------|-----------|------|
| **칸반 미팅 (Kanban Meeting)** | Daily | 15분 이내 | 오른쪽→왼쪽(Right to Left)으로 보드를 훑으며 **완료**에 집중. 사람이 아닌 **일**에 대해 논의 |
| **보충 회의 (Replenishment Meeting)** | Weekly | 20~30분 | 다음에 시작할 일을 우선순위에 따라 선택하여 큐를 채움 |
| **흐름 검토 (Flow Review)** | Bi-weekly | 30분 | 지표를 보고 시스템 자체를 어떻게 개선할지 논의 |

> [!tip] 데일리 미팅이 15분인 이유
> 15분을 넘으면 인지 능력이 저하됨. 데일리 미팅은 **듣는 자리**

### 확장 4가지 Cadences (고성숙도 팀)

| 미팅 | 주기 | 목적 |
|------|------|------|
| **Delivery Planning** | 배포 주기마다 | 실제 납기일 약속, 릴리즈 대상 선정 |
| **Service Delivery Review (SDR)** | Bi-weekly | **고객 관점**에서 서비스 품질 점검 (Flow Review가 내부 지표라면, SDR은 고객 만족도 관점) |
| **Operations Review** | Monthly | 여러 팀/서비스를 가로질러 병목·자원 조율 |
| **Risk Review** | 필요 시 | 블로킹 이슈·외부 의존성의 리스크를 정기 점검 |

## 12. 변화 관리 전략

### 셔틀 외교 (Shuttle Diplomacy)

- 이해관계자를 **개별적으로** 먼저 설득한 후 그룹 킥오프 미팅 개최
- 집단 압력 없이 개인적 동의를 얻어 저항 최소화
- 반대는 주로 특정 관행에 **정체성·자존감·사회적 지위**가 연결된 사람에게서 나온다

### 레드-그레이 다람쥐 전략 (Red-Grey Squirrel Strategy)

- 기존 관행을 즉시 제거하지 않고 새 관행을 **나란히 도입**
- 새 관행이 성공하면 기존 관행이 자연스럽게 도태됨
- 진화적 변화는 견고하고, 강제된 변화는 취약하다

### 포기 단두대 (Abandonment Guillotine)

- **6개월 이상 된 백로그 항목은 자동 삭제**
- 진정으로 중요한 항목은 요청자가 재제출할 것이라는 가정
- 백로그가 통제 불가능하게 커지는 것을 방지

### 신용카드 사기 해결책 (Credit Card Fraud Solution)

- 리스크를 **100% 제거**하려 하지 말고, 경제적으로 허용 가능한 수준으로 **완화**
- XIT 적용: 역량의 30~40%를 견적에 쓰는 것은 2% 리스크를 막기 위한 나쁜 경제학
  → 대신 15일 초과 시 매니저 알림 정책으로 비용 0.5% 미만으로 해결

### 진화적 유물 (Evolutionary Relics)

진화 과정에서 **더 이상 목적을 수행하지 않지만, 제거 메커니즘이 없어 남아있는 관행**을 말한다.

> *기린의 되돌이 후두 신경(Recurrent Laryngeal Nerve)은 뇌에서 후두까지 약 4.5미터를 돌아간다. 직선으로 연결하면 몇 센티미터면 충분한데. 이것이 진화적 유물이다.*

- XIT 사례: 칸반 도입 후 보충 회의에서 제품 매니저들은 ROI 스프레드시트 상위 항목이 아닌, **긴급성과 시의성**에 따라 항목을 선택하게 됨
  - 예: "푸에르토리코 직원 주소 형식 지원" → ROI는 낮지만 다음 달 사무소 오픈 때문에 긴급
  - **지연 비용(Cost of Delay)** 이 ROI를 자연스럽게 대체하기 시작
  - ROI 계산 작업 자체가 **진화적 유물**이 됨
- 보수적·응집력 강한 조직에서는 기존 관행이 유물로 남을 수 있다 → 레드-그레이 다람쥐 전략으로 대응

### 변화 도입 전 — 반대할 사람을 미리 파악하라

칸반 시스템을 설계한 후, 실행 전에 **누가 반대할 것인지, 왜 반대할 것인지**를 미리 분석해야 한다.

| 이해관계자 | 예상 반응 | 이유 |
|-----------|-----------|------|
| **개발자·테스터** | 환영 또는 무관심 | 견적·계획은 그들의 전문 정체성과 무관 |
| **프로그램 매니저** | 저항 가능 | 간트 차트·계획이 PM 정체성과 연결됨 |
| **제품 매니저** | 핵심 장애물 | 견적 없이 ROI 계산 불가 → 우선순위 결정 어려움 |
| **상위 경영진** | 우려 → 조건부 승인 | 검증되지 않은 방식에 대한 불안 |

> 반대는 주로 특정 관행에 **정체성·자존감·사회적 지위**가 연결된 사람에게서 나온다. 셔틀 외교로 개별 동의를 먼저 얻는 것이 핵심이다.

> *"항상 운동선수를 채용하라. 그들은 규율이 있고, 목표를 설정할 줄 알며, 동기부여가 되어 있고, 성과를 측정하고 훈련에 체계적으로 접근한다."*

### 조직 구조 및 제약 조건

- **팀 구성**: 인도 하이데라바드 TCS 아웃소싱 센터 — 개발자 3명, 테스터 3명, 현지 기능 매니저 1명 (총 6인)
- **업무 범위**: 약 80개의 크로스 기능 IT 애플리케이션 유지보수 (재무, 인사, 시설관리, 보안 등 공유 서비스)
- **시차**: 시애틀 ↔ 하이데라바드 12.5~13.5시간 → 실질적으로 주 4일만 협업 가능
- **서비스 유형**: 마이너 기능 업그레이드(변경 요청) + 결함 수정 두 가지

**3가지 핵심 제약**

| 제약 | 내용 | 설정 주체 |
|------|------|-----------|
| **PSP/TSP 방법론 의무 사용** | 개발 프로세스 변경 불가 | 엔지니어링 엑설런스 VP (Jon De Vaan) |
| **추가 예산 없음** | 추가 인력 투입 거부 | 경영진 |
| **거버넌스 규칙** | 15일 초과 작업은 주요 프로젝트로 이관 필요 | PMO |

> [!tip] 역설적 행운
> PSP/TSP 변경 불가 제약이 오히려 "현재 하고 있는 것에서 시작"하는 칸반의 핵심 원칙을 강제했다. 제약이 저항 없는 도입을 가능하게 한 셈이다.

### STATIK 적용 과정

| 단계 | 발견 내용 |
|------|-----------|
| **① 불만족 파악** | 고객: 평균 리드 타임 5.5개월, 납기 준수율 ~0% / 팀 내부: 견적 요구로 역량의 33~40% 낭비, 무작위 작업 투입으로 피로도 극심 |
| **② 수요 분석** | 일반 변경 요청 외 PTC(Production Text Changes)라는 숨겨진 작업 유형 발견. 제출된 요청의 48%가 실제로 납품되지 않음 (26% 폐기, 22% 사건에 의해 추월) |
| **③ 역량 분석** | CFD 분석 결과: 수요가 처리 역량을 크게 초과 → 백로그 월 0.5개월씩 증가. 흐름 효율성 약 8% |
| **④ 워크플로우 모델링** | 단순 Handoff 기준이 아닌 '지식 발견(Knowledge Discovery)' 단계 기준으로 재설계. 변경 요청의 생애주기를 스케치로 시각화하여 병목 지점 파악 |
| **⑤ CoS 식별** | PTC를 Expedite / Fixed Date로 세분화하여 "모두 최우선" 혼란 해소 |
| **⑥ 시스템 설계** | 견적 중단 + WIP 제한 + 계획 회의 → 보충 회의로 전환 |

### 핵심 변화 3가지

#### 변화 1 — 견적 중단 (No Estimates)

견적 활동에 낭비되는 역량을 회수하고, 견적으로 인한 일정 무작위화를 제거하기 위해 개별 견적을 완전히 중단했다.

**견적 중단을 위한 4가지 선택지 비교**

| 선택지 | 설명 | 결과 |
|--------|------|------|
| **① 견적 완전 중단** | SLA 도입으로 대체 | 역량 회수 + 예측성 향상 ✅ **채택** |
| ② 타임 슬라이싱 (Scrum 방식) | 월 8일을 견적·계획에 할당 | 예측성 향상, 역량 회수 없음 ❌ |
| ③ 전담 견적 담당자 지정 | 개발자 1명·테스터 1명 전담 | 방해 감소, 역량 회수 없음 ❌ |
| ④ 순환 견적 담당 | 주 단위로 팀원 간 순환 | 역량 회수 없음 ❌ |

> 역량을 실제로 회수하려면 견적을 **완전히 중단**하는 것만이 유일한 방법이었다.

**견적 대체 방식 — 평균 비용 활용**
- 과거 1년간 엔지니어링 노력 분포 히스토그램 제시 (대부분 3~10일, 평균 약 6일)
- 개별 견적(추측) 대신 **실제 역사 데이터 기반 평균 비용** 사용
- $ROI = \frac{\text{비즈니스 가치}}{\text{평균 비용}}$ 로 우선순위 결정 가능
- 비용이 좁은 범위 내에서 변동한다면, 비용 변동을 무시해도 우선순위 결정에 큰 영향 없음

**거버넌스 규칙 해결 — 신용카드 사기 해결책**
- 문제: 15일 초과 작업은 주요 프로젝트로 이관해야 하는데, 견적 없이 어떻게 판단하나?
- 역사적으로 "너무 큰 작업"은 전체 요청의 **2% 미만**
- 견적에 역량의 30~40%를 쓰는 것은 2% 리스크를 막기 위한 **나쁜 경제학**
- 해결: 개발자가 작업 중 15일 초과 예상 시 → 로컬 매니저에게 알림 → 주요 프로젝트로 이관
- 이 새 정책의 비용: 역량의 **0.5% 미만**

#### 변화 2 — WIP 제한

| 적용 대상 | 내용 |
|-----------|------|
| 개발자 1인당 | 동시 작업 **1건** (PSP에서 이미 권장하던 방식) |
| 테스터 | 동일한 규칙 적용 |
| 개발↔테스트 사이 | **버퍼(크기: 5)** 삽입 — PTC 수신 및 흐름 평활화 목적 |

> 초기 버퍼 크기는 임의로 설정 후 경험적으로 조정. 경험상 **WIP=5가 좋은 출발점**이다.

#### 변화 3 — 계획 중단 (No Planning)

- 월간 계획 회의 폐지 → **주간 보충(Replenishment) 회의**로 대체
- 간트 차트 폐지, 조기 확정(Early Commitment) 폐지
- 백로그는 칸반 시스템에 투입될 때까지 **미확정(Uncommitted)** 상태 유지
- 주간 회의 핵심 질문: *"다음 25일 내 납품을 원하는 항목 3개는 무엇인가?"*
- **25일 SLA**: 평균 엔지니어링 시간 11일 대비 충분한 여유, 기존 140일 대비 획기적 단축

### 변화 구현 방식

- Product Studio 도구에 **저장 프로시저 + DB 트리거**로 WIP 제한 및 자동 이메일 알림 구현
- 6개월 안정화 후 주간 보충 회의도 자동화: 빈 슬롯 발생 시 드라고스에게 자동 이메일 발송 → **슬롯 발생 후 2시간 내 보충**

### 팀 구성 최적화 — 제약 이론(TOC) 적용

안정화 이후 드라고스는 테스터의 여유 역량을 발견하고 인도 팀을 2주간 직접 관찰하여 팀 구성을 조정했다.

| 시점 | 구성 변화 | 처리량 변화 |
|------|-----------|-------------|
| 초기 | 개발자 3 : 테스터 3 | 분기 45건 |
| **1차 조정** | 개발자 4 : 테스터 2 | 분기 **56건** (거의 선형 증가) |
| **2차 조정** (2005년 7월) | 개발자 5 : 테스터 3 | 처리량이 수요를 초과 |

> 개발자만 추가하면 테스트가 병목이 될 위험이 있다. 칸반 시스템에 인력을 추가할 때는 **병목을 만들지 않도록 위치를 신중히 고려**해야 한다.

### 최종 성과

| 지표 | 개선 전 | 개선 후 |
|------|---------|---------|
| 평균 리드 타임 | 5.5개월 | **14일** (92% 감소) |
| 납기 준수율 (25일 SLA) | ~0% | **98%** |
| 처리량 | 월 6건 | **3배 이상** |
| 백로그 | 80건 이상 | **완전 소멸** (2005.11.22) |
| 수상 | — | 2005년 하반기 **엔지니어링 엑설런스 어워드** |

> [!success] 핵심 교훈
> PSP/TSP 방법론 변경 없이, 추가 예산 없이, 하이데라바드 팀이 큰 변화를 인식하지 못한 채 달성한 결과다. **"현재에서 출발"하는 진화적 접근법의 힘.**
>
> 2006년 Robert Bosch(사우스 벤드, 인디애나)에서 Eric Landes가 동일한 결과를 재현 → 드라고스나 저자 없이도 작동함을 검증. 이후 2007년 완전한 **칸반 방법(Kanban Method)** 으로 발전했다.

## 14. 핵심 용어 정리

| 용어 | 설명 |
|------|------|
| **WIP** | Work In Progress. 현재 진행 중인 작업의 수 |
| **리드 타임** | 요청 접수부터 완료까지의 총 시간 |
| **흐름 효율성** | 리드 타임 중 실제 작업 시간의 비율 |
| **커밋먼트 포인트** | 작업 실행을 공식 약속하는 지점 |
| **케이던스** | 반복 이벤트의 리듬/빈도 |
| **지연 비용 (Cost of Delay)** | 작업을 늦게 처리함으로써 발생하는 비용 |
| **CFD** | Cumulative Flow Diagram. 누적 흐름도 |
| **STATIK** | Systems Thinking Approach To Introducing Kanban |
| **PTC** | Production Text Changes. 화면 텍스트 등 경량 변경 작업 |
| **진화적 유물** | 더 이상 목적을 수행하지 않지만 제거되지 않고 남아있는 관행 |
| **Upstream Kanban** | 아직 시작하지 않은 일의 큐를 관리하는 영역 |
| **Downstream Kanban** | 이미 시작된 일의 흐름을 관리하는 영역 |
| **Option Discard Rate** | 아이디어 단계에서 버려지는 작업의 비율 |
| **PTC** | Production Text Changes. 화면 텍스트·설정 파일 등 경량 변경 작업. 개발자 불필요, 테스터만 필요 |
| **진화적 유물** | 더 이상 목적을 수행하지 않지만 제거되지 않고 남아있는 관행 |
| **2-Phase Commit** | 1차(개발 시작 결정) + 2차(배포 직전 고객 약속)의 두 단계로 나뉜 약속 구조 |
| **Little's Law** | Lead Time = WIP ÷ Throughput. 리드 타임을 줄이려면 WIP를 줄여야 함 |
| **85th Percentile** | SLA 기준으로 활용하는 리드 타임 분포 지점. 평균값은 이상치에 왜곡되므로 사용하지 않음 |
| **Replenishment** | 큐 보충. Upstream에서 선택된 항목을 Downstream으로 넘기는 행위 |

## 15. 핵심 인사이트

> [!summary] 전체 핵심 3줄 요약
> 1. **칸반 = 터지는 걸 막는 시스템** — WIP Limit으로 과부하를 방지하고 흐름을 안정화
> 2. **WIP를 줄이면 Lead Time이 줄어든다** — 리틀의 법칙, 사람을 쥐어짜는 게 아님
> 3. **Commitment는 최대한 미뤄라** — 불확실성을 인정하고 마지막 순간에 가장 현명한 결정을

- **"터지는 걸 막는 게 칸반이다"** — 과부하를 방지하여 시스템의 예측 가능성을 회복하는 것이 최우선이다.
- **"시작보다는 끝에 집중하라"** — 새로운 일을 벌이는 것보다 기존 일을 끝내는 것이 가치 전달의 핵심이다.
- **"불만이 없으면 칸반 적용의 명분이 없다"** — 불만족은 칸반 도입의 가장 큰 동기부여다.
- **"칸반은 현재 상태에서 시작한다 (Start with what you do now)"** — 우리 팀의 불만족을 해결해 나가는 진화적 과정이다.
- **"Manage the work, not the people"** — 사람을 통제하는 것이 아니라 작업 흐름 자체를 관리한다.
